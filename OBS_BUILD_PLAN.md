## Integrity design
- **freeze**: forecast.final is the freeze point. At core/swarm.py L288 (after the harness edge block L280-287, BEFORE the best-effort display) obs.on_forecast_final() does two things atomically-from-the-caller's view: (1) appends the forecast.final JSONL line carrying model_probability=result['probability'], market_probability=market_odds, edge, consensus, and record_hash; (2) evidence.freeze_forecast() INSERTs the immutable obs_forecasts row (forecast_id PK, record_hash). outcome is NOT known at this point (resolution happens days later in a different process via settle_resolved). resolution.observed / trade.settle / score.brier are APPENDED as separate events + separate obs_resolutions/obs_scores rows keyed by forecast_id/market_id; they never UPDATE obs_forecasts. AFTER-UPDATE and AFTER-DELETE triggers on obs_forecasts raise(ABORT,'append-only: forecast frozen'). Contrast: the EXISTING forecasts/swarm_forecasts/baseline_forecasts/paper_positions/paper_wallet are mutated in place on resolution (calibration.py:116/127, challenger.py:122, wallet.py:197/200) — obs deliberately mirrors a FROZEN copy so the original-vs-resolution distinction is provable. Test 3 proves a resolve cannot change the frozen record_hash or the JSONL line, and that an UPDATE/DELETE on obs_forecasts is rejected.
- **hash_chain**: Each JSONL line in logs/events/<run_id>.jsonl includes prev_hash = sha256 of the canonical-serialized PRIOR line (genesis prev_hash = 64 zeros). Canonical serialization = json.dumps(sort_keys=True, separators=(',',':')) over the full line object EXCLUDING the prev_hash field is NOT done — instead prev_hash chains over the COMPLETE prior line text exactly as written (including its own prev_hash), so any edit to line N changes the recomputed hash that line N+1 must match. verify_chain(run_id) recomputes sha256 of each stored line and checks the next line's prev_hash; it reports the first index where (a) a line's bytes were tampered (its hash no longer matches the successor's prev_hash), (b) a line was inserted (chain link breaks at insertion), or (c) a line was deleted (successor's prev_hash points to a now-missing hash). Append uses an msvcrt.locking exclusive lock on the file handle + reads the last line's hash immediately before writing so concurrent daemons (loop/predict_today/sameday) chain correctly into per-run files. Test 4 mutates/inserts/deletes a line and asserts detection + correct first-bad index.
- **append_only**: TWO enforcement layers. (1) JSONL files are opened 'a' only; there is no code path that rewrites them; the hash-chain makes any out-of-band rewrite detectable. (2) SQLite obs_ evidence tables get AFTER UPDATE / AFTER DELETE triggers that SELECT RAISE(ABORT,...) on obs_forecasts, obs_trades, obs_scores (the resolved/frozen evidence). obs_resolutions/obs_scores are themselves INSERT-only future appends. CRITICAL isolation: obs creates ONLY new obs_ prefixed tables in polyswarm.db via CREATE TABLE IF NOT EXISTS and installs triggers ONLY on its own tables — it never alters or adds triggers to the live forecasts/decisions/paper_* tables, so journal.py/calibration.py/wallet.py behavior is byte-for-byte unchanged. Connections use PRAGMA busy_timeout=5000 and one-statement commits; journal_mode is left at the DB default (no global WAL switch). The append-only spec violations in the operational tables (5 of them) are accepted as-is and shadowed by the frozen obs mirror rather than 'fixed'.
- **blob_storage**: Content-addressed store logs/blobs/<sha256>. data.fetch stores the raw HTTP body (gamma resp.text per page, gdelt resp.text captured BEFORE the _looks_like_json guard so throttle bodies are kept, wiki r.text) and records raw_hash + blob_ref. llm.call stores FULL system+user prompt and FULL completion text as blobs and records prompt_hash + completion_hash + blob_ref (keeps the heavy text out of the JSONL line while preserving full reconstruction for explain/replay). Identical prompts/payloads dedup to one blob. store_blob writes atomically (tmp+rename) and is a no-op if the hash already exists. Blobs are redaction-scrubbed before write (secret values masked) so criterion 6 holds even for blob contents.
- **redaction**: redact.secret_values() builds a denylist from os.environ of every non-empty value whose KEY matches (?i)(.*_API_KEY|.*_KEY|.*_SECRET|.*_TOKEN) plus the explicit ANTHROPIC_API_KEY/OPENAI_API_KEY/OPENAI_BASE_URL/CHALLENGER_API_KEY. Before ANY line or blob is written, every string field is passed through scrub_str() which replaces those substrings with '***REDACTED***', and config dumps additionally drop secret-NAMED keys entirely. This runs on prompts, completions, tracebacks, params, and the run.start config block. Even though the live box is keyless Ollama (no keys present now), the scrubber reads the LIVE process env at write time, so a key added later is still masked. Test 6 injects fake secret env values, emits them through prompt/config/error paths, then greps every file under logs/ and asserts zero hits.

Also record provider, MODEL_FAST/MODEL_DEEP, DEBATE_ROUNDS, LLM_PROVIDER, seed=null and deterministic=false (no RNG seed, no LLM temperature pinning — forecasts are non-reproducible by design; the only seed in-tree is copula.py's local random.seed(42), irrelevant). All git subprocess calls are guarded so a missing git binary never breaks a daemon pass.

## Risks
- SQLite lock contention on polyswarm.db: three daemons (loop/predict_today/sameday) plus settle already write the single file via short-lived connections with no cross-table transaction. Adding obs_ writes (freeze_forecast on every forecast.final, append_trade on every open/settle) raises 'database is locked' risk for the daemons. Mitigation: PRAGMA busy_timeout=5000, single-statement commits, obs DB write is best-effort (JSONL is canonical, a DB failure is caught and the event still lands in the log). DO NOT enable WAL — switching journal_mode is a DB-wide persistent change that would alter behavior for the running daemons.
- Editing core/agent.py:_call_llm is the single highest-risk change: it is the hottest path AND shared by both swarm agents and the local challenger. The localized refactor converts the three per-branch early `return ...strip()` into (text,usage) assignments + one emit before a final return. A mistake could drop the .strip(), return the wrong branch's text, or raise inside the daemon. Mitigation: preserve the EXACT returned string per branch, wrap the entire obs emit in try/except, and make obs import lazy so a missing obs package can never break a forecast.
- contextvars correctness: _call_llm and size_bet are context-free, so correlation IDs (forecast_id/market_id/agent_id/role) are recovered from a ContextVar stack set by higher hooks. This is safe ONLY because the forecast path is synchronous and single-threaded (fresh Swarm per market, agents looped sequentially). If any future threading/async is added, IDs could leak across markets. Mitigation: push/pop per market and per forecast; reset stack at run.start.
- DB path split-brain: DB_PATH resolves './polyswarm.db' relative to CWD in journal/calibration/wallet; the live DB is polyswarm/polyswarm.db (307KB) and a stray 0-byte polyswarm/harness/polyswarm.db exists. If obs resolves the path differently it would write evidence into a different file than the daemons read. Mitigation: config.resolve_db_path() joins the same DATABASE_URL replace() result to the package root (PKG_ROOT) so obs and daemons always agree.
- Instrumenting gamma/gdelt/wiki must not change return values or alter exception handling. Reading resp.text for blobs must happen without consuming a stream and the gdelt capture must occur BEFORE the _looks_like_json guard. Any emit that raises must be swallowed so the fetcher still returns its data/{} exactly as today.
- Disk growth: storing FULL prompt+completion blobs for every llm.call on a slow CPU loop (plus raw HTTP pages) will grow logs/blobs/ steadily. Content-addressing dedups identical prompts but distinct ones accumulate. Mitigation: flag for a retention/rotation policy; blobs are outside the hash chain so pruning old blobs does not break chain verification (only explain/replay loses old full-text).
- git_sha reproducibility trap: HEAD is real but the tree is DIRTY, so recording only git_sha would be misleading. Must record git_dirty=true AND the code_version content-hash, and must EXCLUDE the six *.orig stale copies from code_version or the hash is unstable/wrong.
- Forecasts are non-deterministic (no RNG seed, no LLM temperature). run.start records this (deterministic=false); replay reconstructs the exact prompts/completions/blend that WERE produced (from blobs) but cannot re-derive a forecast bit-for-bit. This is by-design and must be documented so 'reproducible from run.start' is understood as evidentiary replay, not numeric re-execution.
- Daemon vs pass run boundaries differ per entrypoint: loop.run_once returns a rich summary dict, but predict_today.run_once and sameday.run_once return only a scalar count. run.end counters must be synthesized for the latter from the classify/trade events emitted during the pass, or they will under-report. Also predict_today.daemon inlines settle+predict_one (does not call run_once), so its run boundary must be hooked in the daemon loop, not run_once.
