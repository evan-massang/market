# MIROFISH AUDIT — why stale/empty reports slip through (Phase 0)

The pipeline can silently continue when MiroFish fails, returns old content, or returns
nothing. This documents the current path and the exact defects, before the fix.

## Current path (how it works today)
1. **Report creation** — `harness/mirofish.py::forecast_market(question, …)` drives the
   real `:5001` backend through 7 stages: ontology/generate → graph/build → simulation/
   create → prepare → run → report/generate → GET report. Returns a plain dict
   (`ok`, `crowd_probability`, `report_markdown`, `simulation_id`, `stage_reached`,
   `error`). Honors `max_wait`; on timeout returns the furthest `stage_reached`.
2. **simulation_id** — generated by the backend (`/api/simulation/create`).
3. **Report markdown** — fetched via `GET /api/report/<report_id>` (the dashboard proxy
   also assembles partial `section_*.md` files).
4. **Crowd posts** — `harness/mirofish_signal.py::crowd_signal(sim_id, question)` reads the
   simulated posts for a sim_id; `read_crowd_posts(sim_id)`.
5. **Probability extraction** — `mirofish._extract_probability(markdown, question)` (regex
   + optional LLM) on the report; `crowd_signal` also derives an implied probability.
6. **Save** — `mirofish_signal.save_signal(market_id, sig, market_odds, sim_id)` →
   `mirofish_forecasts` (id, market_id, question, crowd_probability, market_odds, n_posts,
   posts, sim_id, created_at).
7. **Pipeline use** — `predict_today._mirofish_report` appends the report text to the
   swarm's `enrichment` string; `predict_one` does `mf = _mirofish_report(...); if mf: enr += mf`.

## Defects (the 10 audit questions)
1. **Stale reuse — YES (critical).** `mirofish.forecast_market` line 395:
   `pname = project_name or f"poly_{abs(hash(question)) % 10_000_000}"`. The project name is
   a deterministic hash of the QUESTION, so the same question always maps to the same backend
   project, and the `already_completed` paths (lines 338, 517) return the OLD report. A June 13
   report is reused verbatim for a June 16 forecast.
2. **Empty/weak reports pass — YES.** `_mirofish_report` returns "" ONLY when EVERY signal is
   empty (`cp is None and not posts and not report_md`). Any non-empty crumb (incl. a stale or
   wrong-question report) is fed to the swarm.
3. **No freshness check.** Nothing compares `completed_at` to the request start, checks report
   age, or checks the report question against the current market.
4. **No question match.** A report for a different question is accepted as long as it's non-empty.
5. **Min posts / min length not enforced.** A 0-post or 2-char "report" is accepted.
6. **Bet proceeds when MiroFish fails — YES.** An empty MiroFish just omits the enrichment; the
   bet is NOT blocked. There is no `MIROFISH_MODE=required` that withholds the bet.
7. **No status recorded.** The decision/journal stores no mirofish_status / usable / reason /
   report hash / age, so neither the dashboard nor the transcript can say whether MiroFish was
   fresh, stale, or skipped. The transcript shows only "MiroFish crowd: 60% YES".
8. **`save_signal` stores empties.** Empty crowd signals (n_posts=0, crowd_probability=null) are
   written, which is why ~30% of recent rows are empty.

## The fix (this change)
- **Force fresh identity** per market+run+timestamp; never accept `already_completed` unless it
  completed after this request began and matches this market (`MIROFISH_FORCE_FRESH`).
- A **strict result contract** (`MiroFishResult`) + `validate_mirofish_report(result, market)`:
  fresh, non-empty, min-posts, min-length, question-match, valid-probability — else `usable=False`
  with a recorded reason.
- **`MIROFISH_MODE=required|degraded|off`** in `predict_today`: required + unusable → NO BET;
  degraded → continue only if other evidence is strong, marked observe-only; off → skip + log.
- A **stale report is never appended** to the swarm enrichment.
- New **`mirofish_runs`** table records the full status; obs events; dashboard + transcript show
  fresh/stale/failed/used; `python -m harness.mirofish_check` CLI; tests with mocked backend.
- Old records are **marked stale/unusable, never deleted**.
